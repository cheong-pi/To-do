# 검증 체크리스트

## 공통 검증

- [ ] 앱이 실행된다.
- [ ] 콘솔에 치명적인 에러가 없다.
- [ ] 모바일 폭에서 텍스트가 겹치지 않는다.
- [ ] 오늘 화면이 첫 진입 화면이다.
- [ ] 주요 화면은 할일, 캘린더, 단어, 뽀모도로 탭으로 관리된다.
- [ ] 할일 리스트에는 `[수정]` 버튼만 직접 표시된다.
- [ ] 할일 리스트 앞 동그라미 체크로 완료 처리된다.
- [ ] 완료된 할일의 동그라미 체크를 다시 누르면 완료가 되돌려진다.
- [ ] 오늘 요약은 할일 리스트 위에 표시된다.
- [ ] 수정 패널은 사이드 영역이 아니라 선택한 할일 바로 아래에 펼쳐진다.
- [ ] 수정 패널은 제목, 날짜, 시간, 마감, 상태, 메모, 진행률을 직접 수정할 수 있다.
- [ ] `[할일 등록]`은 오늘 할일 제목 옆에 표시된다.
- [ ] 할일 등록 폼은 제목, 날짜, 시간, 마감, 날짜 없는 할일, 메모를 입력할 수 있다.
- [ ] 모바일 폭에서 할일 리스트와 수정 패널이 가로 스크롤 없이 표시된다.
- [ ] 수정 패널에는 시작 버튼이 없다.
- [ ] 진행률 UI는 수정 패널에서만 표시된다.
- [ ] 뽀모도로 UI는 뽀모도로 탭에만 표시된다.
- [ ] 미룸 문구가 사용자를 혼내지 않는다.
- [ ] 기존 데이터 삭제 로직이 없다.
- [ ] 앱 이름이 `잊지 마`로 표시된다.
- [ ] 메인 컬러가 `#FFE680`이다.
- [ ] React 컴포넌트 스타일은 CSS Modules를 사용한다.

## 데이터 검증

- [ ] `createdAt`, `updatedAt`이 저장된다.
- [ ] task status enum이 문서와 일치한다.
- [ ] routine repeat 구조가 문서와 일치한다.
- [ ] 반복 일정은 매일, 요일 반복, 특정 기간, 매월을 지원한다.
- [ ] 사용자가 반복 일정을 직접 설정할 수 있다.
- [ ] 요일 반복은 요일 버튼으로 선택할 수 있다.
- [ ] 반복 일정은 routine, routineRule, taskInstance 구조로 분리된다.
- [ ] 반복 규칙을 바꿔도 과거 taskInstance 기록은 바뀌지 않는다.
- [ ] wordReview 복습 결과가 nextReviewDate를 갱신한다.
- [ ] wordBatch가 20개 학습 후 퀴즈/오답 반복 흐름을 표현한다.
- [ ] 뽀모도로 실행 세션을 서버에 저장하지 않는다.
- [ ] 마지막 뽀모도로 설정만 settings에 저장한다.
- [ ] notification 문서가 할 일 알림과 미루기 재알림을 표현한다.
- [ ] event log가 주요 행동을 기록한다.
- [ ] 스키마 변경 시 migration 문서가 있다.

## 오늘 화면 검증

- [ ] 오늘 날짜 task가 표시된다.
- [ ] 마감 D-5부터 task가 표시된다.
- [ ] 오늘 해당 루틴이 표시된다.
- [ ] 특정 기간에 해당하는 task가 표시된다.
- [ ] 오늘 화면은 최대 15개까지만 표시된다.
- [ ] 좌우 화살표로 어제/내일 할일을 볼 수 있다.
- [ ] 15개 초과 시 시간 정해진 오늘 일정, 직접 오늘로 가져온 일정, 오늘 해당 반복 일정, 마감 가까운 일정, 진행 중, 날짜 없는 할 일 순으로 정렬된다.
- [ ] 리스트에 `시간 있음`, `마감 임박` 같은 내부 분류 문구가 표시되지 않는다.
- [ ] 리스트에 `계획`, `진행 중` 상태 뱃지가 표시되지 않는다.
- [ ] 마감은 D-day 형식으로만 표시된다.
- [ ] 가까운 D-day는 부드러운 빨간 배경으로 표시된다.
- [ ] 미루기 횟수는 표시되지 않고 다음 알림 시간만 표시된다.
- [ ] 시간이 있는 할일은 시간과 알림 시간이 중복 표시되지 않는다.
- [ ] 진행 중 상태라는 이유만으로 파란 배경이 적용되지 않는다.
- [ ] 고정 할일로 등록한 항목은 태그처럼 구분된다.
- [ ] 캘린더 색상 선택은 할일 리스트 배경색이 아니라 캘린더 표시용 데이터로 저장된다.
- [ ] 마감일(D-day)과 진행 기간 시작/종료는 별도 입력으로 구분된다.
- [ ] 날짜 없는 할일을 선택하면 날짜, D-day, 진행 기간, 반복 설정이 비활성화된다.
- [ ] D-day를 선택하면 진행 기간과 반복 설정이 비활성화된다.
- [ ] 반복 없음이면 날짜와 시간만 기본 활성화되고 반복 세부 항목은 비활성화된다.
- [ ] 캘린더 색상은 색상 지정 체크박스를 켠 경우에만 선택 가능하다.
- [ ] 완료된 task는 체크 표시되며 다시 되돌릴 수 있다.
- [ ] 예정/언젠가 할 일은 자동으로 과도하게 표시되지 않는다.
- [ ] `[수정]`을 누르면 해당 할 일의 수정 패널이 열린다.
- [ ] 리스트 앞 동그라미 체크가 done 상태로 바꾼다.
- [ ] 완료된 할일의 동그라미 체크가 planned 상태로 되돌린다.
- [ ] 수정 패널에서 날짜와 상태를 직접 변경할 수 있다.
- [ ] 수정 패널의 상태 선택으로 cancelled 상태를 기록할 수 있다.

## 미룸 검증

- [ ] 1번째 미루기는 10분 뒤로 설정된다.
- [ ] 2번째 미루기는 10분 뒤로 설정된다.
- [ ] 3번째 미루기는 10분 뒤로 설정된다.
- [ ] 4번째 이상은 연기 상태로 전환된다.
- [ ] 연기 상태라도 당일 잔여 일정에 남아 다시 진행할 수 있다.
- [ ] 같은 할 일이 오늘/내일 모두 있으면 오늘 연기 일정은 취소되고 내일 일정은 계획 상태로 남는다.
- [ ] 하루만 있는 할 일을 못하면 날짜가 바뀌어도 연기 상태로 남는다.
- [ ] 미룸/연기/취소는 오늘 총 task 계산에 포함된다.

## 타이머 검증

- [ ] 집중 시간, 휴식 시간, 반복 횟수를 그때그때 선택할 수 있다.
- [ ] 마지막 집중 시간, 휴식 시간, 반복 횟수가 저장된다.
- [ ] 집중 종료 후 `[휴식하기]`, `[멈춤]`이 표시된다.
- [ ] 휴식 종료 후 `[집중 시작]`, `[멈춤]`이 표시된다.
- [ ] 반복 횟수 종료 후 `[1세트 더 반복하기]`, `[멈춤]`이 표시된다.
- [ ] 타이머 종료만으로 task를 done 처리하지 않는다.
- [ ] 뽀모도로 실행 세션은 서버에 저장되지 않는다.

## 진행률 검증

- [ ] 진행률은 `-` 버튼, 숫자 입력창, `+` 버튼으로 입력할 수 있다.
- [ ] `+` 버튼은 10% 증가한다.
- [ ] `-` 버튼은 10% 감소한다.
- [ ] 숫자 입력창은 0-100 범위로 제한된다.
- [ ] 100% 완료 시 원본 일정이 완료 처리된다.
- [ ] 100% 미만 완료 시 원본 일정은 유지되고 남은 퍼센트가 표시된다.

## 단어 복습 검증

- [ ] 사용자가 단어를 직접 입력하지 않아도 출제된다.
- [ ] 기본 단어 세트는 초등-중등 영어 수준 3000개를 목표로 한다.
- [ ] 20개 학습 후 질문/복습 단계로 넘어간다.
- [ ] 틀린 단어가 같은 학습 흐름 안에서 반복 출제된다.
- [ ] 단어 보기 후 뜻을 볼 수 있다.
- [ ] `[알겠음]`, `[헷갈림]`, `[모름]`만 주요 선택지로 제공한다.
- [ ] 알겠음은 복습 간격을 늘린다.
- [ ] 헷갈림은 내일 다시 보여준다.
- [ ] 모름은 오늘 다시 또는 10분 뒤 다시 보여준다.
- [ ] 오늘 단어 수는 settings 값을 따른다.
- [ ] 기본 하루 단어 수는 20개이다.

## 알림 검증

- [ ] 앱이 열려 있을 때 타이머 종료 알림이 동작한다.
- [ ] PWA 설치 가능 상태를 확인한다.
- [ ] 브라우저 알림 권한 요청이 사용자 행동 이후에만 발생한다.
- [ ] 알림 문구는 할 일 내용 중심이다.
- [ ] 미루기 알림은 10분 뒤 재알림을 생성한다.
- [ ] 시작 알림 액션은 진행 중으로 바꾼다.
- [ ] 취소 알림 액션은 task를 취소 상태로 바꾼다.
- [ ] 웹 알림 한계를 사용자에게 과장하지 않는다.

## Firebase 검증

- [ ] Google 로그인 가능
- [ ] 로그아웃 가능
- [ ] 사용자별 데이터가 분리됨
- [ ] Firestore security rules가 userId 격리를 강제함
- [ ] 오프라인/재접속 시 데이터가 크게 깨지지 않음

## 회귀 검증

- [ ] 기존 localStorage 데이터가 있다면 마이그레이션 또는 보존 전략이 있다.
- [ ] 새 버전 배포 후 기존 사용자가 빈 화면을 보지 않는다.
- [ ] PWA service worker 캐시가 새 파일을 반영한다.
- [ ] Firebase 환경 변수가 누락되면 친절한 오류를 보여준다.
